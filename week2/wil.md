# 2주차 WIL (What I Learned)

## 계층형 아키텍처 (Layered Architecture)

스프링은 역할에 따라 계층을 나눠서 개발한다.

| 계층 | 역할 | 비유 |
|------|------|------|
| Controller | HTTP 요청/응답 처리 | 웨이터 |
| Service | 비즈니스 로직 처리 | 주방장 |
| Repository | DB 접근 및 데이터 관리 | 창고지기 |
| Entity | DB 테이블과 매핑되는 핵심 객체 | 원재료 |
| DTO | 계층 간 필요한 데이터만 전달 | 주문서/영수증 |

---

## 패키지 구조

도메인형 구조를 사용한다.  
member, product, order처럼 도메인별로 폴더를 나눠서 관련 파일을 한 곳에 모아두는 방식.  
유지보수가 편하고 새 도메인 추가 시 다른 곳에 영향이 없다.

---

## 주요 어노테이션

| 어노테이션 | 설명 |
|------------|------|
| `@Controller` + `@ResponseBody` | HTTP 요청 처리, 반환값을 응답 바디에 작성 |
| `@Service` | 비즈니스 로직 담당 클래스 |
| `@RequiredArgsConstructor` | `final` 필드 생성자 자동 생성 (의존성 주입용) |
| `@RequestBody` | JSON을 객체로 변환 |
| `@PathVariable` | URL 경로의 값을 변수로 사용 |

## 스프링 빈 & 의존성 주입

스프링 컨테이너가 빈을 관리하고, 필요한 빈을 자동으로 주입해준다.  
매번 객체를 `new`로 생성하는 대신 컨테이너에 등록된 객체를 꺼내 쓰기 때문에 **메모리 효율적**이다.
