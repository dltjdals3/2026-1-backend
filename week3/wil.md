# 3주차 WIL 

## 레포지토리 계층

데이터베이스에 접근해서 CRUD 작업을 수행하는 계층이다.
서비스 계층은 필요한 데이터를 다룰 때 레포지토리를 통해서 데이터를 생성, 조회, 수정, 삭제한다.

| 작업 | 설명 |
|------|------|
| Create | 데이터 생성 |
| Read | 데이터 조회 |
| Update | 데이터 수정 |
| Delete | 데이터 삭제 |

---

## ORM과 JPA

### ORM (Object Relational Mapping)
객체와 관계형 데이터베이스의 테이블을 자동으로 매핑해주는 기술이다.

### JPA (Java Persistence API)
자바 진영의 ORM 기술 표준으로 자바에서 객체를 데이터베이스에 저장하고 관리하기 위한 API다.

---

## 엔티티 매니저

객체를 데이터베이스와 연결해주는 창구 역할을 한다.
SQL을 직접 작성하지 않고도 객체를 데이터베이스에 등록, 수정, 조회, 삭제할 수 있다.

| 메서드 | 역할 |
|--------|------|
| `persist()` | 엔티티 등록 |
| `find()` | 엔티티 조회 |
| `remove()` | 엔티티 삭제 |

---

## 영속성 컨텍스트

JPA에서 엔티티를 관리하기 위해서 만들어져있는 가상 공간
엔티티 매니저를 통해 영속성 컨텍스트에 접근

### 엔티티 생명주기

| 상태 | 설명 |
|------|------|
| 비영속 | 영속성 컨텍스트와 관련 없는 상태 |
| 영속 | 영속성 컨텍스트에 의해 관리되고 있는 상태 |
| 준영속 | 영속성 컨텍스트에서 분리된 상태 |
| 삭제 | 삭제 대상으로 등록된 상태 |

### 영속성 컨텍스트의 장점

**1차 캐시**
한 번 조회한 객체를 영속성 컨텍스트가 내부에 저장해두고 재사용하는 구조
영속성 컨텍스트에 이미 등록(캐시)된 객체를 조회할 때는 데이터베이스를 거치지 않아도 된다.

**동일성 보장**
트랜잭션 안에서는 같은 엔티티를 여러번 조회해도 영속성 컨텍스트에 보관된 같은 객체를 그대로 
반환하므로 항상 동일한 객체임이 보장된다.

**쓰기 지연**
변경 사항들을 모아두었다가 트랜잭션이 커밋되는 시점에 한 번에 묶어서 데이터베이스에 보낸다.
DB와의 네트워크 통신 횟수를 줄여서 성능을 최적화할 수 있다.

**변경 감지**
영속성 컨텍스트가 엔티티의 변경 사항을 자동으로 감지하고, 
트랜잭션 커밋 시점에 변경 내용을 DB에 반영하는 것
변경 감지는 영속성 컨텍스트가 관리하는 영속 상태의 엔티티에만 적용
